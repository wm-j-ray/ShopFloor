## Item 21 — Foreman Tier 1 SKILL.md Files

Five skills written. All in `Skills/platform/`. All tier 1, role: foreman, requires_ai: false. validate-vertical.sh updated to 27-check (then 28 after item 22).

### session-init
The startup orchestrator. Defines the canonical session startup sequence and the rationale for each step's order:

1. halt-monitor (emergency brake first)
2. transaction-manager (restore data consistency before validating the vertical)
3. vertical-registration (validate the vertical contract before anything depends on it)
4. context-index-generator (indexes must be current before role stations assemble)
5. Manifest check → rebuild if missing
6. Session state load/create (infrastructure verified, now load context)
7. Role station assembly (load indexes declared in the active skill's contextFingerprint)
8. Hand to vertical coordinator (Managing Editor for StoryEngine)

**Key design decision:** Session Init is the one place the full platform startup sequence is authoritatively defined. Changes to startup order must be reflected here first.

### halt-monitor
Checks for `platform.halt` at the product root (one level up from the project root — the folder that contains project folders). Three cases: halt present (block and report), halt cleared since last session (log HALT_CLEARED, continue), no halt (silent pass). Karen's experience degrades to Tier 0 silently when halted.

### transaction-manager
Scans `.shopfloor/transactions/` for pending transactions. Re-executes each incomplete operation using idempotent semantics. Handles UNRECOVERABLE operations (logs and continues rather than blocking). Maintains the principle that any write operation in the write-back sequence must be idempotent.

### context-index-generator
Generates context indexes declared in VERTICAL.md. Lazy: regenerates only when any source file's `dateModified` is newer than the existing index's `generatedAt`. Compact metadata extraction defined for two current index types: schema templates (extract schema_type, category, vertical, template_version, type prefix, top-level field names) and role definitions (extract role ID, name, layer, tier, domain, skills, routing triggers). Full content is deliberately excluded — the index is a lookup table.

### rebuild
Full reconstruction from ground truth. Sequence: file registry → object model registry → orphan cross-reference → team-manifest → context indexes (force-regenerated) → skill-registry. Logs `PLATFORM_REBUILT` with counts. Notes what cannot be rebuilt: audit.jsonl (irreplaceable), object model records (level 3), global-registry.json (cross-project, manual re-run of vertical-registration), Karen's content.

## Item 22 — affinity-generator

**Deliverable:** `Skills/platform/affinity-generator/SKILL.md` (v2.0 — v1.0 was discarded)

### Why v1.0 was discarded

v1.0 had five critical gaps:
1. **Invocation architecture undefined.** "Receive from invoking role" with no protocol for how a Tier 3 vertical skill invokes a Tier 1 platform skill mid-session.
2. **Three vague affinity dimensions.** Insufficient criteria for consistent AI scoring.
3. **No staleness model.** Content-changed particles would return stale results indefinitely.
4. **No query mode.** Compute-only meant AI work every time a role needed existing results.
5. **Tier 1 + requires_ai unresolved.** 8K budget is tight; the tension was noted but not architecturally resolved.

### Tier Reclassification: Platform Tier 2

**Decision:** affinity-generator is **Tier 2, not Tier 1.** This is the first instance of a platform Tier 2 skill.

**Reasoning:** The Platform Spec §12 describes Tier 2 as "quality control" and notes "platform rules roles (v1.0: TBD)." affinity-generator is analysis work — not floor management (Tier 1) and not Karen-facing production (Tier 3). It requires AI and significant context. Forcing it into Tier 1 would either require lying about `requires_ai` or hitting the 8K ceiling on any meaningful particle collection. The Platform Spec already anticipated platform Tier 2 with "TBD." This skill fills that slot.

**Platform Spec amendment:** §12 should add: "Foreman may own Tier 2 skills where the analysis is domain-agnostic and crosses vertical boundaries."

### Cross-Skill Invocation Protocol

**New mechanism: `platform_dependencies`** — a new field in a vertical skill's `contextFingerprint`:

```yaml
contextFingerprint:
  platform_dependencies:
    - skill: affinity-generator
      modes: [query, compute]
```

Session-init's station-assembly step (step 7) reads `platform_dependencies` and loads the referenced platform skill's SKILL.md alongside the vertical skill's context. The vertical skill can then invoke the platform skill inline with a structured JSON call block.

**Required amendment:** session-init SKILL.md §7 (station assembly) needs to be updated to handle `platform_dependencies`. Noted in the skill's Notes section.

### Five Affinity Types

Not "three semantic dimensions" but five distinct types with explicit criteria:

| Type | What it finds | Score weight |
|------|--------------|-------------|
| Thematic | Shared story territory / same question about human experience | Highest |
| Tonal | Same emotional atmosphere/register (texture, not topic) | Medium |
| Character/Wound | Similar false belief, failure mode, or psychological pattern | High |
| Imagistic | Recurring imagery, sensory patterns, symbolic vehicles | Medium |
| Structural | Same narrative function potential (inciting incident, dark night, etc.) | Medium (speculative) |

Each type has specific criteria for what makes a description specific enough. "Both involve loss" = insufficient. "Both ask whether grief is something you survive or something you become" = specific.

### Three Invocation Modes

| Mode | Behavior | When to use |
|------|----------|------------|
| `query` | Return known affinities from index; no AI | Fast; tight context; asking about known particles |
| `compute` | Run full AI analysis against unscored candidates | New particle promoted; thin index |
| `smart` (default) | Query first; compute only if results are insufficient | Production default |

### Staleness Model

Each affinity-index entry records `sourceModifiedAt` — the `dateModified` of each particle at computation time. On every invocation, a staleness sweep runs before results are returned. Entries where either particle's current `dateModified` is newer than the recorded `sourceModifiedAt` are marked stale and recomputed.

### NULL_AFFINITY Entries

Pairs evaluated and scored below threshold are recorded as null-affinity entries. This prevents re-evaluation of the same pair on every smart-mode pass. As Karen's particle collection grows, null-affinity entries become the majority — without them, every invocation would re-evaluate the entire corpus.

### Scope

**v1.0:** Project-level. `.shopfloor/affinity-index.json` inside the active project.

**v2+ seeds:** Product-root scope. When multiple projects and verticals exist, the affinity-index migrates to `StoryEngine/affinity-index.json` (product root), and pairs are disambiguated by project UUID. The five affinity types are already vertical-agnostic — no fiction-specific vocabulary. A Readwise highlight and a StoryEngine particle are both `isParticle: true` platform entities; the skill's execution logic does not need to change for cross-vertical computation.
