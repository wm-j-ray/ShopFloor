## Key Design Decisions

**Companion file pattern:** Image and PDF captures write two files: `stem.md` (the browser entry, contains the title heading) and `stem.jpg`/`stem.pdf` (the actual media). CaptureDetailView reads the `.md` file (as it always has) then finds the companion by stem matching. This keeps the browser clean ÔÇö only `.md` files appear ÔÇö while giving the detail view the full content.

**PDF layout:** PDFView manages its own scroll position internally. Wrapping it in a SwiftUI ScrollView creates two competing scroll containers and makes it unusable. The PDF branch uses a plain `VStack` so the note section appears below the PDF viewer without fighting for scroll control.

**ContentType fallback:** `load()` prefers the stored contentType from `.shopfloor/files/[UUID].json` (set by the share extension at capture time). Falls back to `ContentType.from(filename:)` for files that predate the contentType field or were imported externally.

## Build Issue Resolved

After creating the two new `.swift` files, Xcode couldn't find `PDFCaptureView` or `ImageCaptureView`. Root cause: `.xcodeproj` is gitignored and regenerated by xcodegen. The project file hadn't been regenerated since the new files were added.

Fix: `cd App && xcodegen generate`. The `sources: - path: Capture` entry in `project.yml` covers the entire `Capture/` tree, so xcodegen picks up new files on the next generate automatically.
